### Android → HarmonyOS AI 辅助转换（A2H）研发提效

- 鸿蒙端追平主端需要将大量 Android 页面从命令式 View / Activity / Fragment 改写为 ArkTS / ArkUI 声明式范式，纯人工迁移成本高，通用大模型直接生成又容易出现编译失败、状态语义漂移和结果不可验收。
- 借鉴 Spec-Coding 思路，自研 A2H 转换链路：源码事实抽取 → 规格文档 → 实现生成 → 独立验证，采用先文档后代码、审批闸门和白名单产物，降低大模型非预期修改风险。
- 建设 Android → HarmonyOS 映射知识库，将控件、API、生命周期、状态管理（`LiveData` / `StateFlow` → `@State` / `@Link` / `@Prop`）等跨范式映射显式化为可检索、可校验的团队资产。
- 通过 KMP 复用业务逻辑，收敛 AI 需要转换的范围，使模型主要处理 UI 与平台适配层；引入编译校验、UI 还原度验收、设备验收和失败回环。
- 结果：典型中等复杂页面从人工 **2-3 人日**缩短到 **0.8-1.2 人日**，效率提升约 **2.5 倍**；试点页面一次编译通过率达到 **70%+**，人工修正量下降约 **40%**，沉淀可复用的转换 SOP 与映射知识库。

### 客户端性能与稳定性优化专项（Android → HarmonyOS）

- Android 阶段参与 Native Crash / ANR / OOM / 卡顿定位等 Topic，基于 Slardar 聚类、Tombstone、ANR traces、Perfetto、Memory Profiler、btrace / Sliver 等证据完成线上问题归因和复盘沉淀。
- HarmonyOS 阶段作为性能稳定性专项 Owner，统一「线上发现 → 线下归因 → 优化 → 同口径复测」闭环，使用 Slardar、DevEco Profiler、SmartPerf、Frame / Trace 等工具治理冷启动、卡顿丢帧和内存问题。
- 针对启动链路拆解 Launch 阶段耗时，将任务分为首屏必需 / 非必需，推动延迟初始化、异步化、缓存复用和依赖收敛；针对复杂页面通过抓栈与 Trace 定位主线程耗时、布局抖动和列表复用问题。
- 建立每日性能优化日报和同口径复测机制，确保优化收益可对比、问题可追踪、劣化可回滚。
- 结果：在重点场景中，冷启动耗时优化约 **15%-20%**，复杂页面卡顿帧占比下降约 **20%-30%**，内存峰值下降约 **10%-15%**；形成 Android / HarmonyOS 可复用的性能排查手册和复盘模板。

### 鸿蒙「一多」工程实践与 Android 多设备能力推演

- 面向手机、双折叠、三折叠、PAD、横竖屏与连续形变场景，负责多设备适配方案设计与落地，解决早期按机型各写一套、适配口径不一、重复代码多的问题。
- 基于「产品定制层 / 基础特性层 / 公共能力层」分层思路，确立「连续形变用自适应、跨断点用响应式」原则，统一断点、栅格、安全区、状态栏和挖孔区适配策略。
- 在鸿蒙 + KMP 双仓沉淀多设备 UI 适配规范和工具类，针对 KMP 车系页横竖屏切换状态丢失问题设计「进入态保持 + 中转退出」方案，保障切换连续性。
- 将一多理念推演到 Android 折叠屏 / PAD / 横竖屏适配，统一两端多形态适配口径。
- 结果：适配规范覆盖 **20+ 核心页面 / 组件**，重复适配代码减少约 **30%**，多设备问题回归效率提升约 **40%**，并支撑双折叠 **4.79**、三折叠 **4.54** 的体验评分。
